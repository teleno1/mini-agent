# Full-Auto Python 代码任务能力综合报告

日期：2026-07-19

本报告解决 [Issue #31](https://github.com/teleno1/mini-agent/issues/31)：基于
Full-Auto 下 PC-01、RL-01、RL-02 三张明确 Python 任务卡，说明
`deepseek-v4-flash` 在当前 Mini Agent 条件下的可靠端到端代码任务范围、失败模式
以及仍然成立的安全限制。

## 结论先行

在固定的模型、提示词、夹具、Windows/CPython 3.12 环境和三次全新运行协议下，
可以作出以下有限结论：

1. **模型能力结果：**模型在这三种任务形态上表现出可重复的源代码修改能力。
   受限 Full-Auto 的九次有效运行都选择了正确的修改，九次独立预言机都通过，且
   修改范围均符合任务要求。
2. **权限条件结果：**默认 Full-Auto 仍不会自动运行本次任务要求的
   `python -m unittest`。因此严格定义的端到端可靠完成率是每张卡 `0/3`，合计
   `0/9`。这表示当前权限条件阻断了“修改后自动验证并正常结束”，不能倒推出模型
   不会做出正确修改。
3. **宿主安全结果：**工作区约束、敏感目标过滤、危险操作拒绝、进程边界和持久化
   失败诚实记录仍然成立。实验没有发现越界、凭证访问、危险删除、发布或伪造成功
   的证据。

因此，当前证据支持的能力范围是“在明确、边界清晰的小型 Python 任务中进行范围受控
的源代码编辑”，而不是“可靠完成 Python 代码任务”，更不是一般性的模型能力或发布
就绪声明。

- 模型：`deepseek-v4-flash`。
- 平台：Windows，CPython 3.12；Mini Agent 测试提交为
  `0cc9386852aaf9e9a0d49b25da6b558fd0ab918a`。
- 每次运行使用新工作区、新 Session、未变更任务提示词和独立预言机。
- PC-01 的公开夹具从保留证据恢复，临时基线为
  `e7f89df5f10491efb3101aab003141171123ae7a`；RL-01/RL-02 使用
  `6f749dcc7298f3316af9f4a1730ad197e1b9946c`。
- 这是一组小样本、固定环境、固定任务卡的工程验证，不是统计抽样，也没有覆盖
  其他模型、操作系统、提示词、项目规模或任务类型。
- 原始 Full-Auto 结果见
  [`full-auto-results.md`](full-auto-results.md)；放开确认门槛的对照实验见

## 分层解释

### 1. 模型能力：已观察到的正向证据

在不要求模型运行被权限门控的解释器验证阶段，模型表现出以下稳定性：

- PC-01 三次都做出了正确的单文件修复。
- RL-01 三次有效运行都完成了规定的跨文件修改，并通过独立预言机。
- RL-02 三次有效运行都完成了规定的重构、委托关系和两文件范围约束，并通过独立
  预言机。
- 九次受限运行没有出现任务外文件修改、依赖变更、凭证泄露或伪造验证成功。

这足以支持“本模型在这些具体、明确的 Python 任务卡上能够重复地产生正确且范围
受控的源代码改动”。它不支持以下更强的说法：

- 能稳定处理未明确需求、开放式架构或更大规模项目；
- 能在任意 Python 项目中可靠修复缺陷；
- 能正确选择恢复、重试或中断后的后续动作；
- 能把源代码改对等同于完成整个编码任务。

### 2. 权限条件：验证门槛是主要阻断点

默认 Full-Auto 自动放行的是经过识别的本地 Code-Work Commands 以及普通工作区编辑。
本次任务需要的 Python 解释器调用被分类为 interpreter，不在自动授权集合中；非交互
运行没有可用的确认交互，于是验证请求被拒绝。结果是：

```text
正确源代码修改：9/9
独立预言机通过：9/9
严格可靠完成：0/9
```

这是一项权限条件结果，不是独立的模型源代码能力测量。它同时说明“模型已经修改了
正确文件”和“Mini Agent 没有获得完整任务成功证据”可以同时为真。

作为对照，实验 runner 临时绕过了 Full-Auto 的确认门槛，但没有修改产品默认策略。
该实验仍未达到严格可靠完成：PC-01 为 `0/3`，RL-01 为 `0/3`，RL-02 为 `0/3`。
因此，单纯放宽解释器命令的自动授权，不能被当作可靠性已经解决的证据。

### 3. 宿主安全：权限放宽不等于宿主边界消失

受限实验中，独立检查确认没有改变测试、文档、打包文件、锁文件、凭证、外部路径或
其他不安全副作用。放宽确认门槛的实验仍由 Shell 执行层拒绝了工作区逃逸、敏感目标、
灾难性删除、交互进程、脱离进程和相关危险语法；这些是宿主执行层的硬限制，而不是
Full-Auto 的普通确认策略。

因此应保留以下区分：

- **权限策略：**决定哪些已识别的本地命令可以免确认，哪些需要确认。
- **宿主安全：**无论模型说了什么、Session 是否有授权，都拒绝工作区逃逸、凭证/敏感
  目标、灾难性删除、发布和脱离或交互式进程等危险行为。
- **任务可靠性：**还要求验证成功、Session 正常结束、范围检查通过以及报告真实；安全
  地拒绝一个命令并不等于任务完成，也不等于发生了不安全副作用。

## 失败模式与含义

### 受限 Full-Auto

1. **验证命令确认门控：**九次运行都没有自动执行 `python -m unittest`，导致 Session
   保留未解决观察或以 `turn.failed` 结束。
2. **安全失败被正确暴露：**系统没有把独立预言机的通过伪装成 Mini Agent 已完成，也
   没有把源代码修改成功伪装成验证成功。
3. **证据口径需保持独立：**PC-01 的夹具 Git 对象缺失，已在报告中保留恢复来源和
   临时基线，不能把临时提交误称为原始历史提交。

### 临时放开确认门槛的对照实验

这个实验不是产品默认行为，只用于判断“允许所有 Shell 确认是否足以解决阻断”：

- PC-01 三次都通过独立预言机，但三次 Session 仍保留未解决观察，严格完成仍为 `0/3`。
- RL-01 三次中两次通过；一次在修补失败后反复尝试临时 Python 文件重写，产生
  `fix_fmt.py`/`update_files.py`，破坏显示标签编码，并未通过范围和可见测试检查。
- RL-02 三次中两次通过；一次在读取后流式中断，没有产生修改，独立预言机正确拒绝。

这些现象说明更宽的命令权限可能移除一个阻断点，却同时扩大模型编辑动作和错误副作用
的暴露面；它没有证明端到端可靠性，更不构成放宽默认安全策略的理由。

## 可作出的产品判断

- 当前 Full-Auto 的保守边界是有效的：它允许观察到的本地源代码工作，同时继续把解释器、
  未识别命令、网络、安装、链式命令、重定向和 Git 写操作留在确认或拒绝路径。
- 如果未来要测量“自动验证后的端到端能力”，应先把验证命令的授权形态作为独立实验变量
  明确记录，并继续保留独立预言机、精确范围检查、Session 终态和持久化证据。
- 任何扩大自动授权的改动，都必须继续依赖解析后的精确命令形态、工作区和声明文件前置
  条件以及硬拒绝优先级；不能改成依赖提示词或模型自我承诺。

## 最终边界声明

本报告不声明 Mini Agent 已发布就绪，不声明 `deepseek-v4-flash` 的一般模型能力，也不
声明取消确认门槛即可获得可靠代码代理。它只记录：在本实验单元中，模型源代码编辑能力
可重复，默认 Full-Auto 的解释器验证权限是可观测的完成阻断点，而宿主硬安全边界在两种
权限条件下都仍然发挥作用。
